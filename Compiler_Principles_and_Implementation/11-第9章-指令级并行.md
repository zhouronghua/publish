# 9 指令级并行

本章是系列文章的第十章，主要介绍CPU流水线、超标量体系架构等硬件设计，和编译器怎么使能这些功能来减少计算的时钟周期。

9.1 概念

指令级并行是是让一个程序中的多个操作同时执行的方法

指令级并行对原生的顺序程序也能带来并行效果

使能指令级并行的方法：

指令流水线，一起触发一条指令，但多条指令的执行时间可以重叠

超标量体系架构执行：一条指令里面由多条标量指令打包而成触发的并行执行

9.2 热身

对下面的c代码：

![图](images/image_0113.gif)

1 void swap(int v[], int k) {

2     int temp;

3     temp = v[k];

4     v[k] = v[k + 1];

5     v[k + 1] = temp;

6 }

![图](images/image_0114.gif)

有两种汇编编译结果。A1：

1 lw $t0, 0($t1) # reg $t0 (temp) = v[k]

2 lw $t2, 4($t1) # reg $t2 = v[k + 1]

3 sw $t2, 0($t1) # v[k] = reg $t2

4 sw $t0, 4($t1) # v[k + 1] = reg $t0 (temp)

A2：

1 lw $t0, 0($t1)

2 lw $t2, 4($t1)

3 sw $t0, 4($t1)

4 sw $t2, 0($t1)

哪种性能更好？

其实上面两种结果主要体现在第3条指令和第4条指令，是相反的。

要弄清这个问题，还得从CPU的指令流水线说起。

9.3.1 指令流水线的原理

常见的指令流水线的前提是一个指令执行过程中能被切割成好几块，当前主流的做法是切分成5个时钟周期来执行，也称为经典RISC流水线。在这个流水线中一条执行时间是5个时钟周期，但执行5个指令也只需要7个时钟周期，相对不做指令级并行的时候5*5=25个时钟周期而言，并行效果不言而喻。

9.3.1.1 IF

指令获取，Instruction Fetch，从代码段中获取指令。

9.3.1.2 ID

指令解码，Instruction Decode，计算机体系架构设计上，除了软件接口指令集外，最核心的就是微架构，所以一般软件生成的指令，还需要翻译成机器能识别的微指令，这样才能真正执行。

9.3.1.3 EX

执行，Execute。

9.3.1.4 MEM

访问内存，Memory Access。

9.3.1.5 WB

寄存器回写。

有很多处理器的流水线不是固定的，例如 Intel Pentium 4，由于x86是CISC，每个指令的长度本身就不一样，实际实现是指令的执行周期也不完全一样，Pentium 4的指令执行周期有的要7步，甚至最长20步的，但设计原理是类似的Recap13.ppt (live.com)（https://view.officeapps.live.com/op/view.aspx?src=https%3A%2F%2Fcse.hkust.edu.hk%2F~hamdi%2FClass%2FCOMP381-07%2FSlides-07%2FRecap13.ppt&wdOrigin=BROWSELINK）：

![图](images/image_0116.png)

这些的并行的步骤，每个CPU（核）每次只能执行每类小步骤中的一步，不同类的小步骤是可以并发执行的，靠着将指令执行过程中的每个小步骤错开并行起来，就实现了指令流水线的功能。

如果指令的总的执行时间是固定的，那么切分出来的步骤越多，那并行效果就会越好，性能越好。

但是步骤越多，让程序能指令级并行起来，编译器的逻辑就越复杂，这种20步的流水线，估计得被编译器团队给喷死，所以后面x86体系架构再也没有突破20步 :)。

9.3.2 数据冲突

指令流水线能并发执行的前提是指令间没有数据冲突。如果指令I1的输入依赖指令I2的输出，那在I2执行完之前I1是没法执行的，这就是数据冲突的含义。数据冲突太多，就会造成编译器无法生成并行度很高的指令流水线序列，这种情况就会造成指令流水线熄火。

回到上面的例子，如果把$t2寄存器的读指令和写指令放到一起，那这2个指令就会造成指令流水线熄火：

![图](images/image_0117.png)

相反，如果把t0和t2寄存器的读写操作插花式排列开来，如果是2步的流水线，可以完全不熄火，对3步的流水线，最多只会熄火一步，所以后面这种插花的方式性能更好：

![图](images/image_0118.png)

9.4 超标量体系架构

除了指令之间的流水线，超标量体系架构依赖指令内部的多条子指令之间的流水线来达到并行的效果。超标量体系架构其实依赖的是单个处理器中的多个IP（这里的IP不是TCP/IP协议栈里面的ip地址，是intellectual property的简称，是处理器里面可以用来拼装成一个大的处理的积木块，也是可以独立运行的处理单元）相互独立执行来实现的。例如一个VLIW（Very Long Instruction Words，超长指令字）里面可能既有主cpu的操作命令，也有DMA处理器的操作指令。一条GPU的指令里面，可能既有控制指令，也有数据指令。

9.5 寄存器和并行

寄存器导致的依赖分三种（下面的读、写都是相对于寄存器来说的）：

真实依赖，先写后读

lw $t0, 0($t1)
st $t0, 4($t1)

反依赖，先读后写

st $t0, 4($t1)
lw $t0, 0($t1)

输出依赖，连续写

lw $t0, 0($t1)
lw $t0, 4($t1)

除了第一种，后面两种如果寄存器数量足够的情况下，可以通过分配额外的寄存器来消除依赖。
寄存器分配算法让我们尽可能的少用寄存器，但多个变量复用同一个寄存器的做法又会额外注入数据依赖，使相关代码无法并行执行。

以(a + b) + c + (d + e)为例，默认寄存器分配算法结果是这样的：

![图](images/image_0121.gif)

1 LD r1, a

2 LD r2, b

3 ADD r1, r1, r2

4 LD r2, c

5 ADD r1, r1, r2

6 LD r2, d

7 LD r3, e

8 ADD r2, r2, r3

9 ADD r1, r1, r2

![图](images/image_0122.gif)

这样分配的结果，导致这些指令能并行的可能性非常小，仅1/2和6/7这种连续的LD指令可以并行，其他都必须等上一行执行完才能开始执行，9条指令需要7条指令顺序执行的时间。

但我们通过抽象语法树的分析看，d+e的计算，其实和这之前的两次加法也是不相干的，也可以并行；5个变量的LD指令，如果寄存器数量足够的情况下，也可以并行。

9.6 基本块调度

基本块调度（Basic Block Scheduling）也称本地调度（Local Scheduling）。主要使用指令依赖图（有的地方也称为数据依赖图）来进行分析，程序中的每条指令是一个节点，如果节点i1使用节点i0定义的变量，则存在一条边(i0, i1)。指令依赖图IDG中的每条边是一个delay，决定了最终至少需要多少个时钟周期才能完成程序执行。

以下面的程序为例：

![图](images/image_0125.gif)

1 LD R2, 0(R1)

2 ST 4(R1), R2

3 LD R3, 8(R1)

4 ADD R3, R3, R3

5 ADD R4, R3, R2

6 ST 12(R1), R4

7 ST 0(R7), R7

![图](images/image_0126.gif)

假定每条LD/ST指令需要5个时钟周期（除非LD指令紧接着ST指令，并且两个指令操作同一个内存地址，这种情况下ST指令只需要3个时钟周期），每条算术运行需要2个时钟周期，算一下该程序需要多少时钟周期？指令依赖图可以这样画：

![图](images/image_0127.png)

但即使不存在数据依赖，如果多个指令使用同一个资源，也需要排队，所以如果先调度LD R2, 0(R1)的话，需要的时钟周期是5+3+5+2+2+3+1=21个：

![图](images/image_0128.png)

而先调度LD R3, 8(R1)的话，需要的时钟周期是5+5+3+3+1=17个。

![图](images/image_0129.png)

算法描述如下：

![图](images/image_0130.gif)

1 RT = empty reservation table

2 foreach vn ∈  V in some topological order:

3     s = MAX {S(p) + de | e = p→n ∈  E}  // 对所有n的前驱p，求p的执行开始时间和p到n之间的delay，并在其中取最大值，也就是节点n的执行开始时间

4     let i = MIN {x | RT(n, j, x) ∈  RT} in

5         s' = s + MAX(0, i – s)

6         RT(n, j, s') = vn // 在所有可获得的RT资源中，取高度最小的一个资源申请表，来调度n

![图](images/image_0135.png)

实际上全局代码调度相对于一个基本块内的调度要复杂得多，主要涉及代码移动，不安全的决策，额外执行了可能不需要的指令等问题。

后支配（Postdominate）：如果一个节点dst到程序终止的所有路径都要经过节点src，则称为src对dst有后支配关系。

控制流一致性（Control Equivalence）：如果dst支配src，并且src后支配dst，则说明src和dst是控制一致的。

如果代码移动前后对应的位置具有控制流一致性，则这种迁移理论上是安全的。

| 232 | /// ClusterNeighboringLoads - 通过“胶合”将邻近的加载指令聚集在一起。 |

| 234 | /// 如果偏移量之间的距离不远（目标特定）， |
| 235 | /// 它会添加MVT::Glue输入和输出，以确保它们被一起调度并且顺序正确。 |
| 236 | /// 这种优化可能通过提高缓存局部性来使某些目标受益。 |
| 237 | /// ClusterNeighboringLoads - Force nearby loads together by "gluing" them. |
| 238 | /// This function finds loads of the same base and different offsets. If the |
| 239 | /// offsets are not far apart (target specific), it add MVT::Glue inputs and |
| 240 | /// outputs to ensure they are scheduled together and in order. This |
| 241 | /// optimization may benefit some targets by improving cache locality. |
| 242 | void ScheduleDAGSDNodes::ClusterNeighboringLoads(SDNode *Node) { |
| 243 | SDValue Chain; |
| 244 | unsigned NumOps = Node->getNumOperands(); |
| 245 | // 获取最后一个操作数，它应该是一个链（chain）操作数，用于指令之间的依赖。 |
| 246 | if (Node->getOperand(NumOps-1).getValueType() == MVT::Other) |
| 247 | Chain = Node->getOperand(NumOps-1); |
| 248 | if (!Chain) |
| 249 | return; |
| 250 | // 跳过任何有捆绑输入的加载指令。可能存在额外的依赖关系， |
| 251 | // 需要不同的顺序，而且添加的胶合可能会导致循环。 |
| 252 | // Skip any load instruction that has a tied input. There may be an additional |
| 253 | // dependency requiring a different order than by increasing offsets, and the |
| 254 | // added glue may introduce a cycle. |
| 255 | auto hasTiedInput = [this](const SDNode *N) { |
| 256 | const MCInstrDesc &MCID = TII->get(N->getMachineOpcode()); |
| 257 | for (unsigned I = 0; I != MCID.getNumOperands(); ++I) { |
| 258 | if (MCID.getOperandConstraint(I, MCOI::TIED_TO) != -1) |
| 259 | return true; |
| 260 | } |
| 261 | return false; |
| 262 | }; |
| 263 | // 查找其他具有相同链的加载指令。找到从相同基指针加载且偏移量不同的加载指令。 |
| 264 | // Look for other loads of the same chain. Find loads that are loading from |
| 265 | // the same base pointer and different offsets. |
| 266 | SmallPtrSet<SDNode*, 16> Visited; |
| 267 | SmallVector<int64_t, 4> Offsets; |
| 268 | DenseMap<long long, SDNode*> O2SMap;  // Map from offset to SDNode. |
| 269 | bool Cluster = false; |
| 270 | SDNode *Base = Node; |
| 271 | if (hasTiedInput(Base)) |
| 272 | return; |
| 273 | // 此算法需要在找到匹配之前使用次数相对较低，以避免在大型块中无用地增加编译时间。 |
| 274 | // This algorithm requires a reasonably low use count before finding a match |
| 275 | // to avoid uselessly blowing up compile time in large blocks. |
| 276 | unsigned UseCount = 0; |
| 277 | for (SDNode::use_iterator I = Chain->use_begin(), E = Chain->use_end(); |
| 278 | I != E && UseCount < 100; ++I, ++UseCount) { |
| 279 | if (I.getUse().getResNo() != Chain.getResNo()) |
| 280 | continue; |
| 281 | SDNode *User = *I; |
| 282 | if (User == Node || !Visited.insert(User).second) |
| 283 | continue; |
| 284 | int64_t Offset1, Offset2; |
| 285 | // 如果用户节点和基节点不是从相同的基指针加载，或者偏移量相同， |
| 286 | // 或者用户节点有捆绑输入，则继续。 |
| 287 | if (!TII->areLoadsFromSameBasePtr(Base, User, Offset1, Offset2) || |
| 288 | Offset1 == Offset2 || |
| 289 | hasTiedInput(User)) { |
| 290 | // FIXME: 如果地址相同应该是可以的。 |
| 291 | // 但是早期优化应该已经消除了其中一个加载。 |
| 292 | // FIXME: Should be ok if they addresses are identical. But earlier |
| 293 | // optimizations really should have eliminated one of the loads. |
| 294 | continue; |
| 295 | } |
| 296 | if (O2SMap.insert(std::make_pair(Offset1, Base)).second) |
| 297 | Offsets.push_back(Offset1); |
| 298 | O2SMap.insert(std::make_pair(Offset2, User)); |
| 299 | Offsets.push_back(Offset2); |
| 300 | if (Offset2 < Offset1) |
| 301 | Base = User; |
| 302 | Cluster = true; |
| 303 | // 重置UseCount以允许更多匹配。 |
| 304 | // Reset UseCount to allow more matches. |
| 305 | UseCount = 0; |
| 306 | } |
| 307 | if (!Cluster) |
| 308 | return; |
| 309 | // 将偏移量按递增顺序排序。 |
| 310 | // Sort them in increasing order. |
| 311 | llvm::sort(Offsets); |
| 312 | // 检查加载是否足够接近。 |
| 313 | // Check if the loads are close enough. |
| 314 | SmallVector<SDNode*, 4> Loads; |
| 315 | unsigned NumLoads = 0; |
| 316 | int64_t BaseOff = Offsets[0]; |
| 317 | SDNode *BaseLoad = O2SMap[BaseOff]; |
| 318 | Loads.push_back(BaseLoad); |
| 319 | for (unsigned i = 1, e = Offsets.size(); i != e; ++i) { |
| 320 | int64_t Offset = Offsets[i]; |
| 321 | SDNode *Load = O2SMap[Offset]; |
| 322 | if (!TII->shouldScheduleLoadsNear(BaseLoad, Load, BaseOff, Offset,NumLoads)) |
| 323 | break; // Stop right here. Ignore loads that are further away. |
| 324 | Loads.push_back(Load); |
| 325 | ++NumLoads; |
| 326 | } |
| 327 | if (NumLoads == 0) |
| 328 | return; |
| 329 | // 通过添加MVT::Glue输出和输入来聚集加载。 |
| 330 | // 这也确保它们按地址递增的顺序被调度。 |
| 331 | // Cluster loads by adding MVT::Glue outputs and inputs. This also |
| 332 | // ensure they are scheduled in order of increasing addresses. |
| 333 | SDNode *Lead = Loads[0]; |
| 334 | SDValue InGlue = SDValue(nullptr, 0); |
| 335 | if (AddGlue(Lead, InGlue, true, DAG)) |
| 336 | InGlue = SDValue(Lead, Lead->getNumValues() - 1); |
| 337 | for (unsigned I = 1, E = Loads.size(); I != E; ++I) { |
| 338 | bool OutGlue = I < E - 1; |
| 339 | SDNode *Load = Loads[I]; |
| 340 | // 如果AddGlue失败，我们可能会留下一个未使用的胶合值。 |
| 341 | // 这应该不会导致任何问题。 |
| 342 | // If AddGlue fails, we could leave an unsused glue value. This should not |
| 343 | // cause any |
| 344 | if (AddGlue(Load, InGlue, OutGlue, DAG)) { |
| 345 | if (OutGlue) |
| 346 | InGlue = SDValue(Load, Load->getNumValues() - 1); |
| 347 | ++LoadsClustered; |
| 348 | } |
| 349 | else if (!OutGlue && InGlue.getNode()) |
| 350 | RemoveUnusedGlue(InGlue.getNode(), DAG); |
| 351 | } |
| 352 | } |

| 441 | /// CheckForLiveRegDef - 如果指定SUnit的指定寄存器定义破坏了任何“活跃”寄存器， |

| 448 | SmallVectorImpl<unsigned> &LRegs, |

| 461 | /// DelayForLiveRegsBottomUp - 如果为了满足活跃物理寄存器的依赖关系而需要延迟 |
| 462 | /// 调度给定节点，则返回true。如果特定节点是最后一个可调度的节点，则采取必要措施 |
| 463 | /// （例如回溯或克隆）使其可调度。 |
| 464 | /// DelayForLiveRegsBottomUp - Returns true if it is necessary to delay |
| 465 | /// scheduling of the given node to satisfy live physical register dependencies. |
| 466 | /// If the specific node is the last one that's available to schedule, do |
| 467 | /// whatever is necessary (i.e. backtracking or cloning) to make it possible. |
| 468 | bool ScheduleDAGFast::DelayForLiveRegsBottomUp(SUnit *SU, |
| 469 | SmallVectorImpl<unsigned> &LRegs){ |
| 470 | if (NumLiveRegs == 0) |
| 471 | return false; |
| 472 | SmallSet<unsigned, 4> RegAdded; |
| 473 | // 如果此节点会破坏任何“活跃”寄存器，则它尚未准备好调度。 |
| 474 | // If this node would clobber any "live" register, then it's not ready. |
| 475 | for (SDep &Pred : SU->Preds) { |
| 476 | if (Pred.isAssignedRegDep()) { |
| 477 | CheckForLiveRegDef(Pred.getSUnit(), Pred.getReg(), LiveRegDefs, |
| 478 | RegAdded, LRegs, TRI); |
| 479 | } |
| 480 | } |
| 481 | for (SDNode *Node = SU->getNode(); Node; Node = Node->getGluedNode()) { |
| 482 | if (Node->getOpcode() == ISD::INLINEASM || |
| 483 | Node->getOpcode() == ISD::INLINEASM_BR) { |
| 484 | // 内联汇编可能会破坏物理寄存器定义。 |
| 485 | // Inline asm can clobber physical defs. |
| 486 | unsigned NumOps = Node->getNumOperands(); |
| 487 | if (Node->getOperand(NumOps-1).getValueType() == MVT::Glue) |
| 488 | --NumOps;  // Ignore the glue operand. 忽略glue操作数。 |
| 489 | for (unsigned i = InlineAsm::Op_FirstOperand; i != NumOps;) { |
| 490 | unsigned Flags = |
| 491 | cast<ConstantSDNode>(Node->getOperand(i))->getZExtValue(); |
| 492 | unsigned NumVals = InlineAsm::getNumOperandRegisters(Flags); |
| 493 | ++i; // Skip the ID value. 跳过ID值。 |
| 494 | if (InlineAsm::isRegDefKind(Flags) || |
| 495 | InlineAsm::isRegDefEarlyClobberKind(Flags) || |
| 496 | InlineAsm::isClobberKind(Flags)) { |
| 497 | // 检查寄存器定义或earlyclobber寄存器定义。 |
| 498 | // Check for def of register or earlyclobber register. |
| 499 | for (; NumVals; --NumVals, ++i) { |
| 500 | unsigned Reg = cast<RegisterSDNode>(Node->getOperand(i))->getReg(); |
| 501 | if (Register::isPhysicalRegister(Reg)) |
| 502 | CheckForLiveRegDef(SU, Reg, LiveRegDefs, RegAdded, LRegs, TRI); |
| 503 | } |
| 504 | } else |
| 505 | i += NumVals; |
| 506 | } |
| 507 | continue; |
| 508 | } |
| 509 | if (!Node->isMachineOpcode()) |
| 510 | continue; |
| 511 | const MCInstrDesc &MCID = TII->get(Node->getMachineOpcode()); |
| 512 | if (!MCID.ImplicitDefs) |
| 513 | continue; |
| 514 | for (const MCPhysReg *Reg = MCID.getImplicitDefs(); *Reg; ++Reg) { |
| 515 | CheckForLiveRegDef(SU, *Reg, LiveRegDefs, RegAdded, LRegs, TRI); |
| 516 | } |
| 517 | } |
| 518 | return !LRegs.empty(); |
| 519 | } |
| 520 | /// ListScheduleBottomUp - 自底向上列表调度器的主循环。 |
| 521 | /// ListScheduleBottomUp - The main loop of list scheduling for bottom-up |
| 522 | /// schedulers. |
| 523 | void ScheduleDAGFast::ListScheduleBottomUp() { |
| 524 | unsigned CurCycle = 0; |
| 525 | // 释放特殊Exit节点的所有前驱节点。 |
| 526 | // Release any predecessors of the special Exit node. |
| 527 | ReleasePredecessors(&ExitSU, CurCycle); |
| 528 | // 将根节点添加到可用队列中。 |
| 529 | // Add root to Available queue. |
| 530 | if (!SUnits.empty()) { |
| 531 | SUnit *RootSU = &SUnits[DAG->getRoot().getNode()->getNodeId()]; |
| 532 | assert(RootSU->Succs.empty() && "Graph root shouldn't have successors!"); |
| 533 | RootSU->isAvailable = true; |
| 534 | AvailableQueue.push(RootSU); |
| 535 | } |
| 536 | // 当可用队列不为空时，取出优先级最高的节点。如果它尚未准备好，则将其放回队列。 |
| 537 | // 调度该节点。 |
| 538 | // While Available queue is not empty, grab the node with the highest |
| 539 | // priority. If it is not ready put it back.  Schedule the node. |
| 540 | SmallVector<SUnit*, 4> NotReady; |
| 541 | DenseMap<SUnit*, SmallVector<unsigned, 4> > LRegsMap; |
| 542 | Sequence.reserve(SUnits.size()); |
| 543 | while (!AvailableQueue.empty()) { |
| 544 | bool Delayed = false; |
| 545 | LRegsMap.clear(); |
| 546 | SUnit *CurSU = AvailableQueue.pop(); |
| 547 | while (CurSU) { |
| 548 | SmallVector<unsigned, 4> LRegs; |
| 549 | if (!DelayForLiveRegsBottomUp(CurSU, LRegs)) |
| 550 | break; |
| 551 | Delayed = true; |
| 552 | LRegsMap.insert(std::make_pair(CurSU, LRegs)); |
| 553 | CurSU->isPending = true;  // 此SU当前不在AvailableQueue中。 |
| 554 | NotReady.push_back(CurSU); |
| 555 | CurSU = AvailableQueue.pop(); |
| 556 | } |
| 557 | // 所有候选节点由于活跃物理寄存器依赖而被延迟。 |
| 558 | // 尝试通过代码复制或插入跨寄存器类副本来解决。 |
| 559 | // All candidates are delayed due to live physical reg dependencies. |
| 560 | // Try code duplication or inserting cross class copies |
| 561 | // to resolve it. |
| 562 | if (Delayed && !CurSU) { |
| 563 | if (!CurSU) { |
| 564 | // 尝试复制产生这些“昂贵复制”值的节点以打破依赖关系。 |
| 565 | // 如果即使这样也无法解决，则插入跨寄存器类副本。 |
| 566 | // Try duplicating the nodes that produces these |
| 567 | // "expensive to copy" values to break the dependency. In case even |
| 568 | // that doesn't work, insert cross class copies. |
| 569 | SUnit *TrySU = NotReady[0]; |
| 570 | SmallVectorImpl<unsigned> &LRegs = LRegsMap[TrySU]; |
| 571 | assert(LRegs.size() == 1 && "Can't handle this yet!"); |
| 572 | unsigned Reg = LRegs[0]; |
| 573 | SUnit *LRDef = LiveRegDefs[Reg]; |
| 574 | MVT VT = getPhysicalRegisterVT(LRDef->getNode(), Reg, TII); |
| 575 | const TargetRegisterClass *RC = |
| 576 | TRI->getMinimalPhysRegClass(Reg, VT); |
| 577 | const TargetRegisterClass *DestRC = TRI->getCrossCopyRegClass(RC); |
| 578 | // 如果跨寄存器类副本的寄存器类与RC相同，则可以直接复制值。 |
| 579 | // 不要尝试复制定义。 |
| 580 | // 如果跨寄存器类副本的寄存器类与RC不同，则可以复制值，但需要跨寄存器类副本， |
| 581 | // 这很昂贵。 |
| 582 | // 如果跨寄存器类副本的寄存器类为null，则无法复制值。 |
| 583 | // If cross copy register class is the same as RC, then it must be |
| 584 | // possible copy the value directly. Do not try duplicate the def. |
| 585 | // If cross copy register class is not the same as RC, then it's |
| 586 | // possible to copy the value but it require cross register class copies |
| 587 | // and it is expensive. |
| 588 | // If cross copy register class is null, then it's not possible to copy |
| 589 | // the value at all. |
| 590 | SUnit *NewDef = nullptr; |
| 591 | if (DestRC != RC) { |
| 592 | NewDef = CopyAndMoveSuccessors(LRDef); |
| 593 | if (!DestRC && !NewDef) |
| 594 | report_fatal_error("Can't handle live physical " |
| 595 | "register dependency!"); |
| 596 | } |
| 597 | if (!NewDef) { |
| 598 | // 副本问题，这些可能是昂贵的跨寄存器类副本。 |
| 599 | // Issue copies, these can be expensive cross register class copies. |
| 600 | SmallVector<SUnit*, 2> Copies; |
| 601 | InsertCopiesAndMoveSuccs(LRDef, Reg, DestRC, RC, Copies); |
| 602 | LLVM_DEBUG(dbgs() << "Adding an edge from SU # " << TrySU->NodeNum |
| 603 | << " to SU #" << Copies.front()->NodeNum << "\n"); |
| 604 | AddPred(TrySU, SDep(Copies.front(), SDep::Artificial)); |
| 605 | NewDef = Copies.back(); |
| 606 | } |
| 607 | LLVM_DEBUG(dbgs() << "Adding an edge from SU # " << NewDef->NodeNum |
| 608 | << " to SU #" << TrySU->NodeNum << "\n"); |
| 609 | LiveRegDefs[Reg] = NewDef; |
| 610 | AddPred(NewDef, SDep(TrySU, SDep::Artificial)); |
| 611 | TrySU->isAvailable = false; |
| 612 | CurSU = NewDef; |
| 613 | } |
| 614 | if (!CurSU) { |
| 615 | llvm_unreachable("Unable to resolve live physical register dependencies!"); |
| 616 | } |
| 617 | } |
| 618 | // 将未准备好的节点重新添加到可用列表中。 |
| 619 | // Add the nodes that aren't ready back onto the available list. |
| 620 | for (unsigned i = 0, e = NotReady.size(); i != e; ++i) { |
| 621 | NotReady[i]->isPending = false; |
| 622 | // 由于回溯，可能不再可用。 |
| 623 | // May no longer be available due to backtracking. |
| 624 | if (NotReady[i]->isAvailable) |
| 625 | AvailableQueue.push(NotReady[i]); |
| 626 | } |
| 627 | NotReady.clear(); |
| 628 | if (CurSU) |
| 629 | ScheduleNodeBottomUp(CurSU, CurCycle); |
| 630 | ++CurCycle; |
| 631 | } |
| 632 | // 由于是自底向上调度，反转顺序。 |
| 633 | // Reverse the order since it is bottom up. |
| 634 | std::reverse(Sequence.begin(), Sequence.end()); |
| 635 | #ifndef NDEBUG |
| 636 | VerifyScheduledSequence(/*isBottomUp=*/true); |
| 637 | #endif |
| 638 | } |

| 660 | void ScheduleDAGLinearize::ScheduleNode(SDNode *N) { |
| 661 | // 如果节点已经被调度过，则报错 |
| 662 | if (N->getNodeId() != 0) |
| 663 | llvm_unreachable(nullptr); |
| 664 | // 如果节点是EntryToken或被动节点，则不需要转换为MI |
| 665 | if (!N->isMachineOpcode() && |
| 666 | (N->getOpcode() == ISD::EntryToken || isPassiveNode(N))) |
| 667 | // These nodes do not need to be translated into MIs. |
| 668 | return; |
| 669 | LLVM_DEBUG(dbgs() << "\n*** Scheduling: "); |
| 670 | LLVM_DEBUG(N->dump(DAG)); |
| 671 | Sequence.push_back(N);  // 将节点添加到序列中 |
| 672 | unsigned NumOps = N->getNumOperands(); |
| 673 | if (unsigned NumLeft = NumOps) { |
| 674 | SDNode *GluedOpN = nullptr; |
| 675 | do { |
| 676 | const SDValue &Op = N->getOperand(NumLeft-1); |
| 677 | SDNode *OpN = Op.getNode(); |
| 678 | // 如果操作数是glue类型，则将其调度到当前节点之前 |
| 679 | if (NumLeft == NumOps && Op.getValueType() == MVT::Glue) { |
| 680 | // Schedule glue operand right above N. |
| 681 | GluedOpN = OpN; |
| 682 | assert(OpN->getNodeId() != 0 && "Glue operand not ready?"); |
| 683 | OpN->setNodeId(0); |
| 684 | ScheduleNode(OpN); |
| 685 | continue; |
| 686 | } |
| 687 | // 如果操作数已经被调度过，则跳过 |
| 688 | if (OpN == GluedOpN) |
| 689 | // Glue operand is already scheduled. |
| 690 | continue; |
| 691 | // 查找glue操作数的代表用户 |
| 692 | DenseMap<SDNode*, SDNode*>::iterator DI = GluedMap.find(OpN); |
| 693 | if (DI != GluedMap.end() && DI->second != N) |
| 694 | // Users of glues are counted against the glued users. |
| 695 | OpN = DI->second; |
| 696 | // 减少操作数的度数，如果度数为0则调度该操作数 |
| 697 | unsigned Degree = OpN->getNodeId(); |
| 698 | assert(Degree > 0 && "Predecessor over-released!"); |
| 699 | OpN->setNodeId(--Degree); |
| 700 | if (Degree == 0) |
| 701 | ScheduleNode(OpN); |
| 702 | } while (--NumLeft); |
| 703 | } |
| 704 | } |
| 705 | /// findGluedUser - 通过遍历使用链找到glue值的代表用户。 |
| 706 | /// findGluedUser - Find the representative use of a glue value by walking |
| 707 | /// the use chain. |
| 708 | static SDNode *findGluedUser(SDNode *N) { |
| 709 | while (SDNode *Glued = N->getGluedUser()) |
| 710 | N = Glued; |
| 711 | return N; |
| 712 | } |
| 713 | // 线性化调度 - 将DAG线性化，不进行调度优化。 |
| 714 | void ScheduleDAGLinearize::Schedule() { |
| 715 | LLVM_DEBUG(dbgs() << "********** DAG Linearization **********\n"); |
| 716 | // 用于存储粘贴节点。 |
| 717 | SmallVector<SDNode*, 8> Glues; |
| 718 | // 记录DAG的大小（需要调度的节点数）。 |
| 719 | unsigned DAGSize = 0; |
| 720 | // 遍历DAG中的所有节点。 |
| 721 | for (SDNode &Node : DAG->allnodes()) { |
| 722 | SDNode *N = &Node; |
| 723 | // 使用节点ID记录节点的度（使用数量）。 |
| 724 | // Use node id to record degree. |
| 725 | unsigned Degree = N->use_size(); |
| 726 | N->setNodeId(Degree); |
| 727 | unsigned NumVals = N->getNumValues(); |
| 728 | // 检查是否有Glue类型的值，并且该值被使用。 |
| 729 | if (NumVals && N->getValueType(NumVals-1) == MVT::Glue && |
| 730 | N->hasAnyUseOfValue(NumVals-1)) { |
| 731 | SDNode *User = findGluedUser(N); |
| 732 | if (User) { |
| 733 | Glues.push_back(N); |
| 734 | GluedMap.insert(std::make_pair(N, User)); |
| 735 | } |
| 736 | } |
| 737 | // 如果是机器指令，或者不是被动节点，则计入DAG大小。 |
| 738 | if (N->isMachineOpcode() || |
| 739 | (N->getOpcode() != ISD::EntryToken && !isPassiveNode(N))) |
| 740 | ++DAGSize; |
| 741 | } |
| 742 | // 处理粘贴节点，更新其用户节点的度。 |
| 743 | for (unsigned i = 0, e = Glues.size(); i != e; ++i) { |
| 744 | SDNode *Glue = Glues[i]; |
| 745 | SDNode *GUser = GluedMap[Glue]; |
| 746 | unsigned Degree = Glue->getNodeId(); |
| 747 | unsigned UDegree = GUser->getNodeId(); |
| 748 | // 粘贴用户必须与粘贴操作一起调度，因此粘贴操作的其他用户应视为其用户。 |
| 749 | // Glue user must be scheduled together with the glue operand. So other |
| 750 | // users of the glue operand must be treated as its users. |
| 751 | SDNode *ImmGUser = Glue->getGluedUser(); |
| 752 | for (const SDNode *U : Glue->uses()) |
| 753 | if (U == ImmGUser) |
| 754 | --Degree; |
| 755 | GUser->setNodeId(UDegree + Degree); |
| 756 | Glue->setNodeId(1); |
| 757 | } |
| 758 | // 预留足够的空间来存储调度序列。 |
| 759 | Sequence.reserve(DAGSize); |
| 760 | // 从DAG的根节点开始调度。 |
| 761 | ScheduleNode(DAG->getRoot().getNode()); |
| 762 | } |
